Start here because this is the control theory for everything else. Your problem is not “how do we generate screens?” It is “how does a user safely delegate goals to a system that acts across tools?”

The synthesis: what the new UX looks like

The emerging model is:

User intent + user values + current context + app capabilities + task state → generated visual workspace → user feedback/intervention → agent execution → verification → memory/update.

The new UX is not a screen. It is a personal adaptive workspace where:

the user states or implies intent;

the user’s agent understands goals, preferences, constraints, and values;

application agents expose capabilities instead of forcing navigation through apps;

UI fragments are generated into the user’s space;

those fragments inherit common design tokens, spatial rules, motion rules, and accessibility constraints;

the system presents choices visually when visual comparison is faster than text;

the user can redirect, correct, approve, undo, or teach the system;

the system learns, but remains inspectable.

The unboundedness is reconciled by constraints. Generated UI should be bounded by:

Task model — what the user is trying to accomplish.

Data model — what entities, relationships, and states matter.

Capability model — what tools and app agents can actually do.

User model — values, preferences, habits, permissions, expertise, and current cognitive load.

Design grammar — tokens, typography, spacing, hierarchy, motion, layout, depth, and theming.

Context model — device, environment, time, current activity, urgency, and risk.

Control model — approval, undo, explanation, intervention, fallback, and recovery.

Evaluation model — whether the generated UI improved outcome quality, not merely reduced clicks.

Agreed. The workflow should be constructed from orchestration theory, not from an arbitrary list of agents.

The correct abstraction is:

A workflow is a governed reasoning system.
An orchestrator is the mechanism that decides what reasoning operation runs next, what artifact has authority, when evidence is sufficient, when the system must stop, and how discoveries update the higher-level problem definition.

For generated UI/UX, this matters because the design space is unbounded. You do not control it by enumerating every possible UI. You control it by defining the terrain, values, communication rules, design grammar, risk model, and verification gates that any generated UI must respect.

1. Foundational orchestration theory
1. Problem-first control theory

The governing artifact is not a requirements list. It is the problem map.

The attached AI-engineering material explicitly distinguishes three layers: problems define the terrain, values define the direction, and proposals are candidate moves inside that terrain. This means the workflow should not start by asking “what UI should we produce?” It should start by asking “what interaction problem, communication problem, user-control problem, and risk problem does this surface contain?” 

problem-first-ai-engineering

For generated UI, this becomes:

Problem: The user needs to understand, decide, act, redirect, or delegate.

Values: The interface should minimize friction without hiding control.

Proposal: Generate a visual surface, dashboard, choice panel, flow, ambient indicator, or agentic control surface.

The proposal is always challengeable. The problem and values govern it.

2. Values as decision infrastructure

Generated UI cannot be governed by generic rules like “make it clean,” “make it modern,” or “make it intuitive.” Those are too weak. The workflow needs explicit decision axes: simplicity vs expressiveness, automation vs user control, personalization vs predictability, visual density vs cognitive ease, ambient behavior vs explicit visibility, and so on.

The attached book is explicit that philosophy must be written as decision axes with positions and reasoning, because flat values do not resolve tradeoffs. 

problem-first-ai-engineering

For this workflow, the philosophy layer should govern questions like:

When should an agent act automatically?

When must it ask?

When should it show visual evidence?

When should it use text?

When should UI disappear?

When does personalization become loss of control?

When does visual richness become distraction?

When should generated UI preserve application identity versus blend into the user’s common space?

This is the layer that prevents generated UI from becoming arbitrary.

3. Proposal-as-probe theory

A generated UI proposal is not only a candidate design. It is also a probe that reveals missing structure in the problem map.

The book describes proposals as instruments that reveal new problem surfaces, emergent sub-problems, and gaps. The convergence loop terminates only when alignment checks stop producing meaningful surprises. 

problem-first-ai-engineering

That is crucial for HCI because many problems are only visible after a concrete interface exists. A visual proposal may reveal that the user needs a recovery path, that the automation is too opaque, that two values conflict, or that a supposedly “simple” UI hides too much state.

So the workflow should not treat early UI proposals as deliverables. It should treat them as diagnostic instruments.

4. Workflow-first, agent-second theory

Modern agent guidance makes a useful distinction between workflows, where the sequence is controlled by code or explicit structure, and agents, where the model dynamically decides what to do. Anthropic’s guidance recommends starting with the simplest structure and adding autonomy only when it clearly improves outcomes. 
Anthropic

That maps directly to this system:

Use agentic behavior only where the path cannot be predetermined: open-ended research, exploratory visual synthesis, adversarial critique, interaction-model discovery, and context-sensitive proposal generation.

The workflow owns the lifecycle. Agents perform bounded reasoning operations.

6. Creator/evaluator separation

A generator should not certify its own work. The book treats creator/evaluator separation as structural, not optional: creators defend their own assumptions, while fresh evaluators catch drift. 

problem-first-ai-engineering

For generated UI, this means:

The agent that proposes a visual system should not judge whether it preserves user control.

The agent that writes adaptive copy should not judge whether the copy is manipulative.

The agent that designs an invisible interface should not judge whether enough observability remains.

The agent that generates a design grammar should not judge whether it is compatible with the user’s existing space.

Evaluation must be separate, artifact-grounded, and rubric-driven.

7. Risk-proportional process theory

The book separates change risk from verification risk. Change risk concerns blast radius, reversibility, observability, and confidence. Verification risk concerns whether the checks themselves are adequate. 

problem-first-ai-engineering

For UI/UX, this becomes:

A small visual tweak may have low change risk but high communication risk if it changes how the user interprets an option.

An invisible automation may reduce interaction cost but increase verification risk because the user may not notice incorrect behavior.

A personalized interface may improve speed but reduce predictability.

A generated UI fragment may look good but fail compatibility with the user’s broader workspace.

The workflow must scale process to risk, not to artifact size.

8. Three-dimensional verification theory

The book defines three verification dimensions:

Behavioral: Does it work?

Problem: Does it address the terrain?

Value: Does it embody the principles?

It also recommends specialized reviewers because one reviewer checking everything performs worse than separate checks for structure, behavior, problem coverage, and value alignment. 

problem-first-ai-engineering

For generated UI, the equivalent is:

Behavioral: Can the user complete the task?

Problem coverage: Does the UI address the real interaction, communication, control, and context problems?

Value alignment: Does it preserve the intended tradeoffs: low friction, user agency, visual clarity, trust, reversibility, compatibility?

A UI can be behaviorally functional and still be wrong.

9. Event-driven coordination theory

The workflow should not be a fixed linear script. The book recommends routing by results, event-driven coordination, and explicit state machines. A producing agent should report what happened; a routing layer should decide what runs next. 

problem-first-ai-engineering

This is essential for HCI research because research can invalidate the original framing, proposals can reveal missing problems, and risk checks can force deeper evidence gathering.

The orchestrator should route by events:

research_insufficient → run deeper research

problem_map_changed → rerun philosophy extraction

proposal_revealed_new_axis → expand problem map

high_control_risk → require user-control audit

visual_grammar_conflict → run compatibility review

personalization_opacity_detected → require transparency pattern

alignment_passed → generate execution spec or prototype plan

human_value_needed → stop and surface decision

10. Durable-artifact theory

Long-running agents degrade. The book emphasizes short sessions, durable handoffs, decision logs, traceability, and history to prevent re-derivation. 

problem-first-ai-engineering

This aligns with current agent engineering practice. LangGraph emphasizes durable execution and resuming without reprocessing after interruption, while Anthropic’s long-running-agent guidance focuses on preserving state across context windows.

The core loop should be:

→ map problem terrain

→ integrate evidence into problem map
→ define philosophy axes

→ generate proposal variants

→ revise proposal
→ repeat until convergence
→ assess risk
→ define prototype/evaluation plan
→ preserve decisions and handoff

The important part is the loop:

proposal → alignment → discovery → artifact update → proposal revision

This prevents the workflow from converging on the first plausible design. The attached book treats convergence as the point where new checks stop producing meaningful surprises, not as an arbitrary number of review rounds.

14. Correctness model

The workflow should not define “correct answer” as one final UI.

For this space, correctness means:

A proposal is correct enough when it accounts for the mapped problem terrain, embodies the chosen values, communicates appropriately to the user, preserves user control, fits the generated UI grammar, satisfies agent/app contracts, and passes risk-scaled verification without producing new meaningful surprises.

That is the convergence model.

A generated UI is not “right” because it looks good. It is right because it is traceable:

UI element
→ communication purpose
→ user intent
→ problem axis
→ value axis
→ design grammar rule
→ risk policy
→ evaluation result

If that chain breaks, the UI is not governed.

2. The core orchestration theories
A. Ecosystem theory: workflows compose operators, tools, clients, and conventions

VALUES.md defines the repository as an ecosystem of small specialized pieces:

operators do operator-shaped work
clients wrap external services
tools perform small mechanical tasks
workflows compose operators, tools, and clients
conventions encode rules

That matters for HCI because a generated UI system should not become one giant “UX agent.”

The correct architecture is:

HCI workflow
  → calls HCI operators
  → uses lean app/context/design-token clients
  → follows HCI conventions
  → writes durable artifacts
  → routes by risk, currentness, and results

For generated UI, this means:

The workflow owns lifecycle and routing.

The operators perform bounded reasoning operations.

The clients expose external app/user/context/design systems.

The conventions define the design and safety rules.

The artifacts preserve state, decisions, and traceability.

The generated UI is only the output surface. The workflow is the controlling system.

B. Routing theory: AGENTS.md is a topology catalog, not just documentation

AGENTS.md is not simply a list of agents. It is a routing table that decides which orchestration topology applies.

The key pattern is:

Classify the work
→ select strategy
→ select flavor
→ dispatch the correct workflow/operator
→ preserve artifacts
→ audit the process

For HCI, this means the system must first classify the requested work:

HCI request shape	Correct route
“Research this design frontier”	HCI research workflow
“Design a new generated UI strategy”	Full adaptive-interface workflow
“Review this existing UI/proposal”	HCI audit / review-first workflow
“Explore visual directions”	HCI prototype workflow
“Ship a bounded UI behavior”	HCI implementation pipeline
“Coordinate multiple UI/application-agent changes”	HCI feature-development workflow
“Reduce known ergonomic risk”	HCI risk-reduction workflow

That is different from simply dispatching “UX agent” or “UI agent.”

C. Flavor theory: strategy and risk posture are separate

The Work Manager flavor files are important. They show that the same workflow can run under different risk postures:

manager-max
manager-pragmatic
manager-hackerman

These are not different workflows. They are different risk appetites.

Translated to HCI:

Flavor	Use case	Risk posture
hci-max	persistent user-space UI, agent delegation, high-stakes workflows, cross-app automation	All major gates must return LOW. No opacity/control residuals.
hci-pragmatic	bounded UI improvement, internal tool, local visual refinement	Accept narrow MEDIUM residuals only when they do not affect control, safety, accessibility, or trust.
hci-prototype / hci-hackerman	visual exploration, speculative interaction model, fast answer-seeking	Can tolerate mess during exploration, but must end in a dossier with risk profile and spawned follow-up work.

This is critical for generated UI. The workflow may be the same, but a UI fragment that automates a financial decision requires a different risk posture than a throwaway visual storyboard.

D. Prior-state theory: before proposing, understand the existing surface

This is the biggest thing my earlier answer underweighted.

This is software-oriented in the repo, but the theory transfers directly to HCI.

In software, prior state means: what code, tests, contracts, entrypoints, duplicate systems, and lifecycle already exist.

In HCI, prior state means:

Generated UI without prior-state understanding is dangerous because it can overwrite the user’s mental model. It can make a “better” UI that is ergonomically wrong because it fails to account for what the user already knows, sees, expects, and trusts.

E. Per-surface risk theory: risk is not global

conventions/risk-profile.md is a major expansion. It says the risk profile is not a single score. It is a per-surface scored map. A single work unit can be lean in one surface and exhaustive in another.

That should become a rule for generated UI:

Do not assign one risk level to the whole interface. Score each interaction surface.

For HCI, a “surface” can be:

choice panel
confirmation flow
ambient notification
generated dashboard
cross-app command surface
personalized recommendation
undo/recovery path
agent progress view
visual comparison layout
app-agent UI fragment
voice/gesture/text multimodal entrypoint

Each surface gets its own risk profile and mode:

LOW     → lean generation/evaluation
MEDIUM  → lean with explicit callouts
HIGH    → exhaustive research, review, and testing

This avoids the bad global decision: either over-process everything or under-process the dangerous part.

F. LOW-only gate theory

The repository repeatedly uses LOW-only gates for pipeline-callable checks. MEDIUM and HIGH are not “acceptable residuals” in those contexts. They trigger revision, remediation, rerun, or decomposition.

That is a strong orchestration theory:

A gate is not advice if downstream work depends on it.

For HCI, LOW-only gates should apply to:

user-control risk
transparency risk
accessibility risk
app-agent contract risk
irreversible-action risk
personalization opacity risk
safety/privacy risk
cross-app automation risk

For example:

If generated UI hides why an option changed → personalization opacity risk is not LOW.
If the user cannot undo or intervene → control risk is not LOW.
If app-agent capability semantics are unclear → agent-contract risk is not LOW.
If visual state looks complete but omits uncertainty → communication risk is not LOW.

A generated UI should not advance just because it looks polished.

G. Currentness and staleness theory

models.zip has a much stronger staleness model than the book alone.

Prior reviews, audits, LOW verdicts, risk profiles, and assumptions become stale after substantive changes unless exact identity/currentness can be proven. The repo uses concepts like target identity, content hashes, current audit bundles, and stale LOW invalidation.

For generated UI, this is extremely important.

A prior UI review can become stale when:

the user’s context changed
the user corrected the system
an app capability changed
a design token changed
the app-agent contract changed
the task lifecycle changed
a new modality was added
a new UI fragment was introduced
the user’s values changed
the surrounding workspace changed

A generated UI system should not say “this was approved before” unless it can prove the relevant state is still current.

This creates a new HCI artifact:

hci-currentness-record.md

It should record:

user-state version
design-token version
app-agent contract version
capability manifest version
layout/theme version
known user preference version
last review target identity
what changed since review
which prior findings are stale

That prevents stale ergonomic assumptions from becoming current UI.

H. Prototype theory: when framing is unclear, the deliverable is a dossier

workflows/build-prototype.md is not just “build a prototype.” It has a different theory:

When you do not yet know what to build, do not pretend you can gate a proposal.
Explore quickly.
Then stabilize the answer.
Then produce a dossier.

For HCI/generated UI this is essential. Many visual and interaction questions cannot be answered by research alone.

The prototype workflow should be used when:

the visual grammar is unknown
the interaction state space is unclear
the user feedback loop is not understood
the app-agent integration surface is unknown
the common-space theming problem is unresolved
the generated UI needs taste/feel exploration
the problem is too large to split safely

But the prototype branch/mockup is not the authority. The dossier is.

The HCI prototype dossier should include:

answer.md
visual-evidence/
interaction-recordings/
risk-profile.md
challenges.md
spawned-tickets.md
branch-disposition.md
design-grammar-findings.md
user-feedback-findings.md
app-agent-contract-findings.md

This fits your SVG/composite/visual exploration direction. Visual drafting can be exploratory, but the output must preserve what was learned.

I. Risk-reduction theory: pay down ergonomic debt between work units

workflows/risk-reduction.md introduces a pattern that should transfer almost directly.

In software:

coverage gap
behavioral ambiguity
duplicate systems
language fragmentation
brittleness
lifecycle invisibility

become targeted risk-reduction work.

In HCI:

unclear user state
ambiguous visual semantics
duplicate app surfaces
modality fragmentation
hidden lifecycle transitions
stale personalization
unobservable automation
inconsistent theming
missing recovery paths

should produce risk-reduction tickets.

A key rule from the repo: one item = one axis × one surface.

So not:

Improve generated UI consistency

Instead:

Reduce duplicate-surface risk for task-status presentation across Calendar, Tasks, and Email fragments.

Or:

Reduce lifecycle-visibility risk for delegated booking flow by making pending, blocked, approved, and completed states visually persistent.

And success requires re-scoring the risk profile.

J. Structured question theory: user input is an artifact, not a casual interruption

conventions/agent-questions-and-session-graph.md is an important expansion.

Sub-agents do not ask the user directly. They emit structured question artifacts. The root orchestrator surfaces the question, records the answer, and resumes the blocked work.

For HCI this becomes a theory of user feedback:

The user’s corrections, overrides, values, and scope decisions should be durable structured artifacts.

Generated UI should not merely ask:

Do you like this?

It should produce structured feedback opportunities:

This layout assumes speed matters more than explanation.
A: keep speed-first
B: show more explanation
C: make this decision contextual
D: never automate this class of choice

The answer updates:

user-value model
personalization policy
communication model
design grammar
risk profile
decision log

That is how feedback becomes usable by future agents.

K. Process-tree theory: the process itself must be auditable

process-tree-auditor, workflow-process-auditor, and workflow-execution-violations.md establish a deeper rule:

A workflow’s claim of success is not proof that it followed the required process.

For generated UI, this means the system should be able to prove:

research happened before proposal
current-state map existed before proposal
risk profile existed before design
evaluation criteria existed before UI generation
creator and evaluator were separate
user-value questions were routed correctly
stale reviews were not reused as current
HIGH/MEDIUM gate failures were not accepted as residuals

This is especially important if AI generates UI dynamically. The user may see only the final surface. But the system needs internal provenance proving that the surface was generated under the right constraints.

3. Translation of software concepts into HCI concepts

This is the central bridge.

models.zip concept	Software meaning	HCI / generated-UI meaning
Existing-state risk profile	Understand touched code before proposing	Understand the user’s current ergonomic state before generating UI
Coverage inventory	What behavior is already tested	What user states, choices, and feedback paths are already represented
Lifecycle map	Process start/end/state transitions	User task lifecycle, engagement state, interruption/resume paths
Entrypoints	CLI/API/jobs/events	Voice, gesture, text, app event, sensor, notification, direct manipulation

Cross-language trace	Contracts across Python/TS/etc.	Contracts across apps, agents, modalities, UI fragments, design systems
Language fragmentation	Implicit contracts across languages	Semantic fragmentation across modalities/apps
Change-path entropy	Same change touches many places	Same UI/meaning rule must be updated across many surfaces
Lifecycle visibility	Can the process be drawn?	Can the user understand where they are and what the agent is doing?
Coverage gap	Behavior untested	User cannot see, verify, or correct an important state
Behavioral ambiguity	Intended behavior unclear	User intent, system action, or UI meaning unclear
Blast radius	Scope of breakage	Number of apps/tasks/decisions affected by generated UI or automation

Push-vs-pull coupling	Push into common interface instead of pulling from uncontrolled source	App agents should expose capability/state contracts instead of user agent scraping app-specific UI

This is where “prior state” becomes ergonomics.

In generated UI, prior state is not a technical afterthought. It is the user’s current lived interface.

5. New HCI risk axes

These should be added to the workflow as the HCI analogue of conventions/risk-profile.md.

Axis	LOW	MEDIUM	HIGH
Visual/communication coverage gap	All important states, choices, and consequences are visible or inspectable	Some states require explanation or secondary inspection	User cannot see material state, uncertainty, or consequence
Intent ambiguity	User intent is clear from explicit action or reliable context	Intent is plausible but partly inferred	System acts on weak or conflicting intent
Interaction blast radius	Affects one local surface	Affects adjacent surfaces/apps	Affects many apps, permissions, automations, or irreversible tasks
Lifecycle visibility	Start, pending, blocked, delegated, completed, failed, and resumed states are clear	Some lifecycle stages are implicit	User cannot tell where they are or what the agent is doing
Duplicate-surface count	One canonical representation	One primary plus known legacy/alternate surface	Multiple surfaces represent the same task differently
Modality fragmentation	Voice/text/touch/gesture/visual semantics align	One modality has special-case behavior	Modalities conflict or create different meanings
Personalization opacity	User can understand, override, and reset adaptation	Adaptation is partly explainable	User cannot tell why UI or choices changed
Feedback/recovery gap	User can correct, undo, retry, or teach the system	Recovery exists but is indirect	No clear intervention or recovery path
App-agent contract ambiguity	Capabilities, permissions, state, and UI payloads are explicit	Some implied contract remains	App agent behavior is opaque or underspecified
Theme/space incompatibility	Surface inherits common-space tokens, density, motion, and depth	Minor incompatibility	Surface feels foreign, conflicting, or semantically inconsistent
Ergonomic brittleness markers	No repeated confusion/correction/workaround	One or two signs	Repeated correction, abandonment, workaround, or explainability patching
Change-path entropy	Rule can be changed in one place	Small predictable propagation	Same semantic change must be edited across many apps/fragments
Trust/accountability gap	User can see who/what acted and why	Partial accountability	Action provenance is hidden

These are not checklist decoration. They decide the mode of the workflow.

8. Retrofits

The repo’s review_first and plug_existing_review modes matter for HCI.

Many generated UI systems will not start from scratch. They will retrofit existing apps, dashboards, flows, and surfaces.

Retrofit workflow:

existing UI / app-agent / design-system surface
→ review-first audit
→ existing ergonomic state profile
→ currentness check
→ stale finding classification
→ proposal
→ risk gates
→ consolidation or generated UI integration

Retrofit rules:

Do not trust an old UX audit as current.
Do not trust an old design-system review if tokens changed.
Do not trust prior user feedback if the task context changed.
Do not trust app-agent contracts if capabilities changed.
Do not merge generated UI into a common space until duplicate surfaces are mapped.

A retrofit is not a normal greenfield design. It is a currentness and compatibility problem.

10. The key strategic correction

The previous answer treated the workflow too much like:

research HCI material
→ define theory
→ create HCI agents
→ produce generated UI strategy

The repository points to a stronger architecture:

route by work shape
→ select flavor
→ map prior/current state
→ score per-surface risk
→ research only the axes that matter
→ generate proposal against current state + problem map + values
→ run LOW-only gates where downstream safety depends on them
→ prototype only when framing is genuinely unknown
→ preserve decisions, questions, reviews, and currentness
→ reduce accumulated ergonomic risk between features

That is the right theory for unbounded generated UI.

The user experience is not just generated. It is orchestrated under constraints.

The central design question becomes:

What current user-space state, risk profile, communication objective, app-agent contract, and design grammar must be true before this UI is allowed to appear?

That is the missing bridge between models.zip and HCI.
